# API-контракт ядра LectureLog

Тонкий документ для интеграторов (прежде всего platform-api). Описывает флоу и
инварианты, которых **нет** в машинной OpenAPI-спеке. Схемы запросов/ответов и
коды статусов здесь **не дублируются** — они в `docs/openapi.json`.

## Источник правды

`docs/openapi.json` — единственный источник правды для генерации типизированного
клиента в platform-api. Файл закоммичен и проверяется CI-джобой `openapi`
(`git diff --exit-code`): если код роутов/схем изменился, а контракт не
перегенерирован, сборка краснеет. Регенерация: `python scripts/export_openapi.py`.

Этот документ покрывает то, что вне OpenAPI: межсервисные флоу, исходящий вебхук,
два endpoint'а MinIO, контур HMAC-доверия и автономные режимы работы.

## Флоу загрузки больших исходников

Большие медиа не грузятся через тело `POST /tasks`, а кладутся напрямую в MinIO
по presigned-ссылке:

1. `POST /uploads` с `{filename}` → ответ `{key, url, expires_in}`. Ключ имеет
   форму `uploads/<uuid>/<safe-filename>` (имя файла санитизируется до basename).
2. Клиент кладёт файл в MinIO `PUT`-запросом по выданному presigned `url`.
3. `POST /tasks` с полем `s3_key` (= тот самый `key`) и опционально `media`
   (`audio` | `video`, по умолчанию `audio`).

**Инвариант безопасности `s3_key`.** Ключ обязан начинаться с `uploads/` и не
содержать traversal-сегментов (`..`). Иначе `POST /tasks` отвечает `400`. Это
защита от IDOR (подстановка чужого ключа бакета) и выхода за пределы `uploads/`.

`POST /tasks` принимает **ровно один** источник из набора `audio | video |
video_url | s3_key`; любое другое количество — `400`.

## Два способа забрать результат

Результат лекции хранится в MinIO **папкой отдельных объектов** под префиксом
`results/<id>/output/...` (см. «Папочная раскладка результата и structure.json»);
единый ZIP **не хранится** — он собирается на лету при скачивании (нет дублирования,
лекции живут бессрочно).

- `GET /tasks/{id}/result` — собирает ZIP **на лету**: листит объекты под
  `results/<id>/`, скачивает их во временный каталог и отдаёт ZIP клиенту (внутри —
  `output/...`, включая `output/structure.json`). Временный каталог чистится после
  отдачи. Работает **всегда**, в том числе без public-endpoint. Это дефолтный,
  автономный путь. Если объектов под префиксом ещё нет — `404` (результат не готов).
- `GET /tasks/{id}/result-url` — presigned `GET` на ZIP. Ядро собирает ZIP на лету
  (как `/result`), заливает его **временным** объектом `results-tmp/<id>/<uuid>.zip`
  и выдаёт presigned-ссылку (с override-заголовками attachment + `application/zip`,
  срок жизни 1 час). Доступен **только** при заданном `S3_PUBLIC_ENDPOINT`; иначе
  `409`. Нужен, когда ссылку отдают прямо в браузер клиента, минуя ядро. Временные
  объекты `results-tmp/` чистятся lifecycle-правилом MinIO (Expiration 1 день) и при
  `DELETE /tasks/{id}`.

Аналогичная развилка для загрузки: presigned PUT из `POST /uploads` требует
public-endpoint, иначе `409`.

## Папочная раскладка результата и structure.json

Результат раскладывается в MinIO отдельными объектами под префиксом
`results/<id>/output/`:

- `output/конспект.md` — markdown-конспект (Obsidian);
- `output/<media_kind>/NN-slug.ext` — медиа-фрагменты по секциям (`audio`/`video`);
- `output/slides/slide-NN.png` — слайды;
- `output/structure.json` — нейтральное дерево результата для платформы.

`result_path` задачи хранит **префикс** `results/<id>/` (не ключ ZIP).

`structure.json` — машиночитаемое дерево, отвязанное от Obsidian-специфики:

```jsonc
{
  "source": {"title": null, "kind": "<media_kind>", "duration": "<end последней секции>"},
  "sections": [                          // верхний уровень = тема (домен Topic)
    {"title": "<Topic.title>",
     "subtopics": [                      // вложенный уровень = подтема (домен Section)
       {"title": "<Section.title>",
        "media": {"kind": "<media_kind>", "start": "<start>", "end": "<end>",
                  "key": "results/<id>/output/<media_kind>/NN-slug.ext"},  // полный ключ MinIO
        "slide_keys": ["results/<id>/output/slides/slide-NN.png", "..."], // полные ключи MinIO
        "content_md": "<markdown>"}                                       // markdown, не html
     ]}
  ]}
}
```

Поля `media.key` и `slide_keys` — это **реальные ключи объектов MinIO** (та же
формула, что у залитых объектов), поэтому платформа может скачать любой фрагмент/слайд
напрямую (или через presigned). Секция без медиа -> `media: null`; без слайдов ->
`slide_keys: []`. `source.duration` — `end` последней секции (или `null`, если секций нет).

**Делёж ядро/платформа.** Ядро отдаёт нейтральное дерево + объекты; рендеринг,
навигацию и собственное представление строит платформа. Способы забрать результат —
см. «Два способа забрать результат».

## Удаление задачи (идемпотентно)

`DELETE /tasks/{id}` инкапсулирует физику хранения: платформа не знает ключей и
схемы ядра, а говорит «удали задачу X». Ядро удаляет свои объекты MinIO
(`results/<id>/...`, временные `results-tmp/<id>/...` и связанный исходник
`uploads/<...>`, если он сохранён при создании через `s3_key`) и строку задачи в БД.

**Идемпотентность.** Платформа ретраит запрос при обрыве соединения, поэтому
повтор на уже удалённую или вовсе неизвестную задачу возвращает `204`, а не
`404`/`500`. Каждый шаг чистки — no-op при отсутствии данных.

Эндпоинт доступен и в автономном (standalone) режиме ядра — он не зависит от
платформенной обвязки. Он находится за тем же внутренним HMAC-контуром доверия
платформа<->ядро, что и остальные мутирующие эндпоинты (`POST /tasks`,
`POST /uploads`): инбаунд-подпись отдельно не вводится.

## Два endpoint'а MinIO

Один MinIO доступен по двум адресам:

- `S3_INTERNAL_ENDPOINT` (обязателен) — адрес внутри docker-сети; по нему ядро
  скачивает/загружает объекты сервер-сервер.
- `S3_PUBLIC_ENDPOINT` (опционален) — внешний хост, который подставляется в
  presigned-ссылки для прямого доступа клиента/браузера. Без него presigned-URL
  наружу не выдаются (`409` на `/uploads` и `/result-url`).

## Исходящий вебхук (не в OpenAPI by design)

OpenAPI ядра описывает **входящие** запросы. Исходящий вебхук — это запрос,
который ядро само отправляет на платформу, поэтому в спеку ядра он не попадает
намеренно. При необходимости его можно задокументировать на стороне platform-api
(в т.ч. через OpenAPI callbacks её собственной спеки).

- **Включение.** Только при заданном `PLATFORM_CALLBACK_URL`. Без него вебхук не
  отправляется — платформа узнаёт статус поллингом `GET /tasks/{id}`.
- **Метод/адрес.** `POST` на `PLATFORM_CALLBACK_URL`, `Content-Type:
  application/json`, короткий таймаут, best-effort (сбой отправки логируется и
  глушится, обработку задачи не ломает).
- **Тело (тонкое).** `{task_id, status, error, error_code}`. Ключи `error` и
  `error_code` присутствуют всегда (`null`, если статус не ошибочный) — тело
  стабильно. `error_code` — машинный код ошибки (`rate_limit` | `bad_input` |
  `internal`), `null` при неошибочном статусе; платформа использует его для
  решения о retry (`rate_limit` повторяем, `bad_input` — нет). Сериализация
  детерминированная: UTF-8, компактный JSON, `sort_keys=True`,
  `ensure_ascii=False` (кириллица в `error` не эскейпится).
- **Подпись.** Заголовок `X-Webhook-Signature` = HMAC-SHA256 от **байтов тела**
  (ровно тех, что ушли в POST) с ключом `LECTURELOG_WEBHOOK_SECRET`, hex.
  Платформа сверяет подпись своим экземпляром секрета.

## HMAC-контур доверия между сервисами

Общий секрет `LECTURELOG_WEBHOOK_SECRET` известен ядру и платформе. Ядро
подписывает им тело вебхука, платформа проверяет. Подписываются ровно отправляемые
байты, поэтому любое расхождение в сериализации ломает проверку — отсюда жёсткий
контракт кодировки тела (см. выше).

## Автономные режимы

Ядро работает без платформенной обвязки за счёт опциональных переменных:

- **Без `PLATFORM_CALLBACK_URL`** — вебхук выключен; статус и `usage` забираются
  поллингом `GET /tasks/{id}`, результат — через `GET /tasks/{id}/result`.
- **Без `S3_PUBLIC_ENDPOINT`** — presigned-ссылки не выдаются (`409` на `/uploads`
  и `/result-url`); загрузка исходников и выдача результата идут через ядро.

В обоих случаях поведение остального API не меняется — это деградация удобства,
а не функциональности.
